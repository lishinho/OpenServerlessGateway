# OpenServerlessGateway (OSG) 开源中间件架构设计文档

**版本**：V1.0
**项目定位**：轻量级、高可扩展、云原生适配的Serverless网关开源中间件，解决微服务/Serverless架构下接口标准不统一、权限管控分散、流量控制粒度粗、研发交付效率低等核心问题，实现流量统一入口、能力收敛标准化、研发流程提效的核心目标。
**适配场景**：云原生Serverless平台（Knative/OpenFaaS/自建）、主流微服务框架（Spring Cloud/Dubbo/SOFAStack）、传统单体应用向微服务/Serverless迁移场景
**核心特性**：无状态部署、多协议转换、插件化鉴权、细粒度流量管控、热部署极速发布、低代码配置、插件化扩展

> 📖 **详细文档导航**：[docs/README.md](docs/README.md) - 包含核心设计文档、模块文档、部署指南等完整文档

## 一、项目背景与核心问题

### 1.1 行业痛点
随着云原生和Serverless架构的普及，企业研发过程中普遍面临以下问题：
1. **架构拆分后的管理混乱**：微服务/Serverless模块拆分数量激增，导致接口协议不统一、跨模块对接成本高，问题排查效率低；
2. **基础能力重复开发**：各业务模块独立实现鉴权、限流、降级、日志监控等能力，研发资源浪费，且能力标准不统一；
3. **流量管控能力不足**：通用网关仅支持粗粒度的接口/IP限流，无法满足业务级的灰度引流、流量回放、防刷等精细化需求；
4. **研发交付效率低下**：前后端联调依赖人工对接，接口发布流程繁琐，模块迭代上线周期长；
5. **架构迁移成本高**：传统单体/微服务应用向Serverless迁移时，存在协议不兼容、流量接入复杂等问题，迁移难度大。

### 1.2 项目核心目标
针对上述痛点，OpenServerlessGateway（OSG）作为**Serverless架构与微服务架构的统一流量入口**，实现以下核心目标：
1. **能力收敛**：将鉴权、限流、降级、协议转换、日志监控等基础能力收敛到网关层，业务模块无需重复开发；
2. **标准统一**：制定统一的接口协议、权限模型、流量管控标准，实现跨模块/跨架构的标准化对接；
3. **研发提效**：提供自动化接口文档、Mock调试、CI/CD流水线对接、热部署等能力，将接口对接/发布效率提升50%以上；
4. **精细化管控**：支持业务级的细粒度流量控制、灰度引流、流量回放、防刷，提升系统稳定性和业务灵活性；
5. **低侵入适配**：采用插件化、无状态设计，支持主流微服务/Serverless框架，实现业务模块"零代码/低代码"接入；
6. **云原生兼容**：支持Docker/K8s容器化部署、Serverless平台弹性伸缩，适配云原生基础设施。

### 1.3 核心设计思想
1. **无状态设计**：网关自身不存储业务状态，所有状态数据依托外部存储（Redis/数据库），支持水平扩展；
2. **分层解耦**：采用分层架构设计，各模块职责单一，通过标准化接口交互，实现模块间解耦；
3. **插件化扩展**：基于SPI机制实现核心能力的插件化，支持自定义鉴权、协议、流量控制插件，满足个性化业务需求；
4. **配置化驱动**：核心能力（协议转换、限流、灰度、降级等）均通过配置中心实现动态配置，无需重启网关即可生效；
5. **热部署与极速发布**：基于类加载器隔离实现插件/配置的热部署，发布耗时控制在秒级；
6. **关注分离**：网关层负责通用基础能力，业务模块专注于业务逻辑，实现架构上的关注分离。

## 二、整体架构设计

OSG采用**分层架构+插件化引擎**的设计模式，整体分为**接入层、核心能力层、引擎层、基础支撑层**，底层对接主流微服务/Serverless平台，上层提供标准化的运维/开发入口，整体架构遵循**高内聚、低耦合、可扩展、高性能**的设计原则。

### 2.2 架构分层职责说明

#### （1）接入层
作为网关的**统一流量入口**，负责接收所有外部请求，实现协议的初步解析和转发，核心职责：
- 支持HTTP/HTTPS、OpenAPI、移动端（APP/小程序）等主流流量接入；
- 支持自定义协议扩展（基于插件化），满足个性化业务协议需求；
- 实现请求的初步校验（请求格式、必选参数），过滤非法请求；
- 提供自动化接口文档入口（Swagger/OpenAPI 3.0），支撑前后端联调。

#### （2）核心能力层
网关的**核心能力载体**，所有通用基础能力均在此层实现，且全部采用**插件化设计**，支持动态加载/卸载，核心职责：
- 封装所有通用基础能力，业务模块无需重复开发；
- 提供标准化的能力接口，支持插件化扩展；
- 接收引擎层的调度，完成请求的处理和转发。

#### （3）引擎层
网关的**大脑核心**，无状态设计，负责网关的整体调度、配置管理、插件生命周期管理，核心职责：
- 实现配置的动态解析和热更新，无需重启网关；
- 基于SPI机制实现插件的加载、初始化、销毁等生命周期管理；
- 实现请求的路由转发和泛化调用，对接后端服务；
- 实现全链路上下文的创建、透传和管理，支撑鉴权、流量管控的精细化；
- 实现统一的日志采集、监控指标上报，支撑问题排查和系统监控。

#### （4）基础支撑层
网关的**底层依赖**，所有组件均采用**可替换、可扩展**设计，适配主流开源组件，核心职责：
- 为网关提供存储、配置、服务发现、监控告警等基础能力；
- 屏蔽底层组件的差异，为上层提供标准化的调用接口；
- 支持组件的无缝替换，满足不同企业的技术栈选型需求。

## 三、核心模块设计

OSG的核心能力层采用**插件化模块设计**，每个模块独立实现单一能力，通过引擎层的标准化接口交互，支持动态扩展和热部署，核心模块包括**流量管控模块、通用鉴权模块、协议转换模块、研发提效模块**，各模块设计如下：

### 3.1 流量管控模块

#### 3.1.1 模块定位
解决通用网关流量管控粒度粗的问题，实现**业务级细粒度的流量管控**，支持限流、降级、重试、灰度引流、流量回放、接口防刷等能力，是提升系统稳定性和业务灵活性的核心模块。

> 📖 **详细设计**：[限流方案设计](docs/design/rate-limiting.md) | [灰度发布设计](docs/design/gray-release.md)

#### 3.1.2 核心功能设计
| 功能点       | 设计说明                                                                 | 实现方式                     |
|--------------|--------------------------------------------------------------------------|------------------------------|
| 细粒度限流   | 支持IP/接口/用户/业务维度的限流，支持分钟/秒级粒度，支持动态配置阈值       | 基于Redis+令牌桶/漏桶算法    |
| 自动降级     | 支持接口级/业务级的自动降级，触发降级后转发至兜底服务/返回兜底数据         | 基于配置中心的降级规则+心跳检测 |
| 智能重试     | 支持接口级的可配置重试，支持指定异常类型/返回码重试，避免无效重试           | 基于配置的重试规则+幂等性校验 |
| 灰度引流     | 支持按用户/IP/业务属性（如行业、商家类型）的灰度引流，支持流量比例调整     | 基于配置的灰度规则+流量转发   |
| 流量回放     | 支持请求的全量/增量录制，记录请求体/返回体/上下文/操作时间，支持离线回放   | 基于日志/消息队列的录制+回放引擎 |
| 接口防刷     | 支持IP/用户的接口防刷，支持动态配置访问频率阈值，触发后阻断/限流           | 基于Redis的滑动窗口算法      |
| 并发控制     | 支持接口级的可配置线程池，实现核心接口与普通接口的并发隔离                 | 基于自定义线程池+动态配置    |

#### 3.1.3 核心设计亮点
- 所有限流/降级/灰度规则均通过**配置中心动态配置**，无需重启网关，实时生效；
- 支持**多维度规则的组合**，满足复杂的业务流量管控需求；
- 流量回放支持**离线分析和自动化测试**，提升回归测试效率；
- 所有流量管控操作均**记录日志和监控指标**，支撑问题排查和效果评估。

### 3.2 通用鉴权模块

#### 3.2.1 模块定位
解决微服务/Serverless架构下权限管控分散、重复开发的问题，实现**标准化的统一鉴权体系**，支持RBAC/ABAC两种主流权限模型，实现权限的精细化管控和全链路上下文透传。

> 📖 **详细设计**：[鉴权方案设计](docs/design/authentication.md)

#### 3.2.2 核心功能设计
| 功能点           | 设计说明                                                                 | 实现方式                     |
|------------------|--------------------------------------------------------------------------|------------------------------|
| 多权限模型支持   | 原生支持RBAC（基于角色）/ABAC（基于属性）权限模型，支持插件化扩展自定义模型 | 基于接口抽象+插件化实现      |
| Session治理      | 支持分布式Session管理，实现Session的共享和失效控制                       | 基于Redis的分布式Session     |
| 权限点映射       | 将前端菜单/按钮/URI映射为标准化的权限点，实现权限的统一管理               | 基于数据库的权限点元数据存储 |
| 全链路上下文透传 | 将鉴权信息（用户ID/角色/权限点）封装到上下文，透传至下游所有服务           | 基于ThreadLocal+HTTP头透传   |
| 免登录鉴权       | 支持测试接口/内部接口的免登录鉴权，提升研发效率                           | 基于配置的免登规则+IP白名单  |
| 动态Token鉴权    | 支持接口级的动态Token鉴权，Token自动刷新，提升接口安全性                   | 基于JWT+Redis的Token管理     |

#### 3.2.3 核心设计亮点
- 鉴权逻辑与业务逻辑**完全解耦**，业务模块只需通过上下文获取鉴权信息，无需实现鉴权逻辑；
- 权限点采用**标准化元数据管理**，支持批量导入/导出/修改，提升权限管理效率；
- 上下文透传支持**跨进程/跨模块**，实现全链路的鉴权信息共享；
- 所有鉴权操作均**记录审计日志**，支撑权限审计和问题排查。

### 3.3 协议转换模块

#### 3.3.1 模块定位
解决微服务/Serverless架构下协议不兼容的问题，实现**多协议的统一转换和泛化调用**，是实现业务模块"零代码"接入网关的核心模块，支持主流RPC协议与HTTP协议的互转。

> 📖 **详细设计**：[HTTP转RPC自动转发设计](docs/design/http-to-rpc.md)

#### 3.3.2 核心功能设计
| 功能点           | 设计说明                                                                 | 实现方式                     |
|------------------|--------------------------------------------------------------------------|------------------------------|
| 多协议互转       | 原生支持HTTP/HTTPS ↔ Dubbo/SOFA/gRPC/TR，支持插件化扩展自定义协议         | 基于协议解析器+插件化实现    |
| 泛化调用         | 支持对后端RPC服务的泛化调用，网关无需依赖业务接口包，实现"零代码"接入       | 基于动态代理+反射机制        |
| API注册与映射    | 支持标准化的API注册，实现HTTP接口与后端RPC服务的一键映射                 | 基于配置中心/数据库的元数据存储 |
| 自定义编排       | 支持多后端服务的接口编排，将多个RPC接口拼接为一个HTTP接口，减少前端调用次数 | 基于配置的编排规则+执行引擎  |
| 报文标准化       | 实现请求/返回报文的标准化封装，解决前后端协议不统一的问题                 | 基于配置的报文模板+数据映射  |

#### 3.3.3 核心设计亮点
- 泛化调用实现**网关与业务模块的解耦**，业务模块升级无需重启网关；
- API注册与映射支持**配置化驱动**，无需编写代码，实现"低代码"接入；
- 报文标准化支持**自定义模板**，满足不同业务的报文格式需求；
- 协议解析器采用**插件化设计**，支持自定义协议的快速扩展。

### 3.4 研发提效模块

#### 3.4.1 模块定位
解决研发过程中前后端联调难、接口发布慢、问题排查效率低的问题，实现**研发流程的标准化和自动化**，是提升研发交付效率的核心模块。

#### 3.4.2 核心功能设计
| 功能点               | 设计说明                                                                 | 实现方式                     |
|----------------------|--------------------------------------------------------------------------|------------------------------|
| 自动化接口文档       | 基于API注册元数据，自动生成Swagger/OpenAPI 3.0接口文档，支持在线调试       | 基于Swagger核心+元数据解析    |
| 接口Mock调试         | 支持接口级的Mock配置，返回自定义Mock数据，实现前后端并行开发               | 基于配置的Mock规则+数据返回  |
| CI/CD流水线对接      | 支持与Jenkins/GitLab CI等主流CI/CD工具对接，实现接口的自动化发布和测试     | 基于WebHook+流水线模板       |
| 统一错误码体系       | 实现前后端统一的错误码体系，支持一级/二级错误码，实现错误码的标准化管理     | 基于数据库的错误码元数据存储 |
| 全链路问题排查       | 支持基于TraceID的全链路日志查询、指标追踪，实现问题的快速定位             | 基于链路追踪+ELK日志分析     |
| 上传下载标准化       | 实现文件上传下载的标准化接口，支持文件流自动转换，实现权限管控下的文件传输   | 基于分布式文件存储+权限校验  |

#### 3.4.3 核心设计亮点
- 自动化接口文档与**API注册元数据实时同步**，无需人工维护，保证文档的准确性；
- Mock调试支持**动态配置**，无需重启网关，实现前后端并行开发，提升研发效率；
- 错误码体系支持**前后端监控串联**，实现错误码的统一排查和管理；
- 全链路问题排查支持**TraceID一键追踪**，实现问题的快速定位，提升排查效率。

## 四、核心特性实现

### 4.1 无状态部署与水平扩展
OSG采用**完全无状态设计**，网关自身不存储任何业务状态和配置数据，所有状态数据依托外部存储（Redis/数据库），配置数据依托配置中心（Nacos/Apollo）：
- 支持**无限水平扩展**，可根据流量大小动态增加/减少网关实例；
- 支持**容器化部署**（Docker/K8s），适配云原生基础设施；
- 支持**Serverless平台部署**，实现网关的弹性伸缩，降低资源成本。

### 4.2 热部署与极速发布
OSG基于**类加载器隔离**和**SPI插件化机制**实现插件/配置的热部署，核心实现：
- 采用自定义类加载器加载插件，插件更新时重新创建类加载器，实现插件的热加载/卸载；
- 配置中心采用**监听机制**，配置变更时实时推送至网关实例，实现配置的热更新；
- 插件/配置的发布耗时控制在**秒级**，实现极速发布，无需重启网关，无服务中断。

### 4.3 插件化扩展机制
OSG采用**SPI（Service Provider Interface）** 实现核心能力的插件化扩展，为所有核心能力定义标准化的接口，第三方开发者只需实现接口并配置插件，即可实现自定义能力的扩展：
1. 网关内置**插件管理引擎**，负责插件的加载、初始化、销毁等生命周期管理；
2. 开发者只需实现对应能力的**标准化接口**，并在`META-INF/services`目录下配置插件全类名；
3. 插件管理引擎会自动扫描并加载插件，支持**动态加载/卸载**，无需重启网关；
4. 支持**插件优先级配置**，实现多插件的有序执行。

**支持扩展的能力类型**：自定义协议解析器、自定义鉴权插件、自定义流量控制插件、自定义日志插件、自定义监控插件等。

### 4.4 低代码/零代码接入
OSG通过**配置化驱动**和**泛化调用**实现业务模块的"低代码/零代码"接入，核心接入流程：
1. 业务模块在**注册中心**完成服务注册；
2. 开发者在**配置中心/网关控制台**配置API元数据（协议类型、接口路径、后端服务映射等）；
3. 网关通过**泛化调用**对接后端服务，无需依赖业务接口包；
4. 完成配置后，网关自动生成**自动化接口文档**，支持在线调试和Mock。

**接入成本**：业务模块无需修改任何代码，只需完成服务注册和配置，即可实现网关接入，接入耗时控制在**分钟级**。

### 4.5 云原生适配
OSG深度适配云原生技术栈，支持**Docker/K8s容器化部署**和**Serverless平台弹性伸缩**，核心适配能力：
1. 提供**官方Docker镜像**，支持一键部署；
2. 提供**K8s部署yaml文件**，支持StatefulSet/Deployment部署，支持服务发现和负载均衡；
3. 支持**Knative/OpenFaaS**等主流Serverless平台，实现网关的弹性伸缩，根据流量大小自动增加/减少实例；
4. 支持**云原生服务网格（Istio）** 集成，实现网关与服务网格的能力互补；
5. 支持**分布式链路追踪（SkyWalking/Pinpoint）**，实现全链路的监控和追踪。

## 五、部署与接入设计

### 5.1 部署方式
OSG支持**多种部署方式**，适配不同企业的基础设施选型，核心部署方式包括：

#### （1）容器化部署（推荐）
- 提供官方Docker镜像，支持一键构建和部署；
- 提供K8s部署yaml文件，支持StatefulSet/Deployment部署，支持服务发现、负载均衡、滚动更新；
- 支持K8s Operator，实现网关的自动化运维（扩缩容/升级/故障恢复）。

#### （2）Serverless平台部署
- 支持Knative/OpenFaaS等主流Serverless平台，实现网关的弹性伸缩；
- 无需关注服务器配置和资源管理，根据流量大小自动扩缩容，降低资源成本。

#### （3）传统虚拟机部署
- 提供jar包部署方式，支持JDK8+/JDK11+；
- 支持单机/集群部署，集群部署时通过注册中心实现服务发现和负载均衡。

### 5.2 技术栈依赖
OSG采用**Java语言开发**（主流微服务开发语言，生态丰富），核心技术栈如下，所有依赖均为**开源主流组件**，无闭源依赖：

| 技术栈类型       | 核心组件                                                                 |
|------------------|--------------------------------------------------------------------------|
| 核心框架         | Spring Boot/Spring Cloud Starter（轻量级、易扩展）                       |
| 配置中心         | Nacos/Apollo/ETCD（可替换）                                             |
| 注册中心         | Eureka/Nacos/Consul（可替换）                                           |
| 存储组件         | Redis（限流/会话/缓存）、MySQL/PostgreSQL（配置/元数据）（可替换）        |
| 监控告警         | Prometheus/Grafana（指标）、ELK（日志）、AlertManager（告警）（可替换）   |
| 链路追踪         | SkyWalking/Pinpoint/Zipkin（可替换）                                     |
| 容器化/编排      | Docker/K8s/Knative/OpenFaaS                                              |
| 构建工具         | Maven/Gradle                                                             |

### 5.3 标准化接入SOP
OSG为业务模块提供**标准化的接入SOP**，实现"低代码/零代码"接入，接入流程分为**4个阶段**，全程无需修改业务模块代码：

#### 阶段1：接入准备
1. 业务模块在**注册中心**完成服务注册（微服务/Serverless函数）；
2. 确认业务模块的协议类型（Dubbo/SOFA/gRPC/HTTP等）；
3. 准备接口元数据（接口名、方法名、参数类型、返回类型等）。

#### 阶段2：网关配置
1. 在**网关控制台/配置中心**配置API元数据，实现HTTP接口与后端服务的映射；
2. 配置基础能力（限流/降级/鉴权/Mock等），根据业务需求选择开启；
3. 配置协议转换规则，实现前端协议与后端协议的互转。

#### 阶段3：联调测试
1. 通过网关**自动化接口文档**进行在线调试，验证接口的可用性；
2. 开启**Mock调试**，实现前后端并行开发；
3. 进行灰度引流测试，验证流量管控规则的有效性。

#### 阶段4：发布上线
1. 配置**灰度引流规则**，逐步将流量切至网关；
2. 监控网关指标和日志，排查潜在问题；
3. 全量切流后，完成业务模块的网关接入。

## 六、扩展能力规划

OSG作为开源中间件，提供**丰富的扩展能力**，支持通过插件化、配置化的方式实现个性化需求，核心扩展能力规划包括：

### 6.1 插件化扩展
基于SPI机制实现**全维度的插件化扩展**，支持第三方开发者自定义插件，实现个性化能力：
- **协议扩展**：自定义协议解析器，支持企业内部私有协议；
- **鉴权扩展**：自定义鉴权插件，支持企业内部的统一鉴权体系（如OAuth2.0/SAML）；
- **流量控制扩展**：自定义流量控制插件，支持企业内部的精细化流量管控规则；
- **日志监控扩展**：自定义日志插件/监控插件，对接企业内部的日志/监控平台；
- **存储扩展**：自定义存储插件，对接企业内部的分布式存储系统。

### 6.2 多平台适配
支持**主流微服务/Serverless/云平台**的适配，实现网关的跨平台部署和使用：
- 微服务框架：Spring Cloud/Dubbo/SOFAStack/ServiceComb；
- Serverless平台：Knative/OpenFaaS/AWS Lambda/阿里云函数计算/腾讯云SCF；
- 云平台：阿里云/腾讯云/华为云/AWS/Azure；
- 服务网格：Istio/Linkerd/Conduit。

### 6.3 可视化能力扩展
后续将开发**网关可视化控制台**，实现网关的可视化配置、运维和监控：
- **配置可视化**：支持API元数据、限流/降级/鉴权/灰度等规则的可视化配置；
- **运维可视化**：支持网关实例、插件的生命周期管理，支持动态加载/卸载插件；
- **监控可视化**：支持网关指标、日志、链路追踪的可视化展示，支持自定义监控面板；
- **告警可视化**：支持告警规则的可视化配置，支持告警信息的实时推送和展示。
